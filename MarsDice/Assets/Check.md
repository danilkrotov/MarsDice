# Чеклист порядка боя (по текущему коду)

Ниже — что происходит **до фаз**, **внутри фаз** и **после каждой фазы**. Порядок фаз задаётся списком `battleActions` на `BattleController` (в типичной сцене: **Восстановление энергии → Пополнение щита → …**; при подключении — **Нанесение урона**).

---

## 1. Запуск боя и раунды

1. **`BattleController.Start`** запускает корутину `PlayTurnSequence`.
2. **Первый кадр** — `yield return null`, чтобы успели отработать `Awake`/`Start` у юнитов и модулей.
3. Внешний цикл **`while (!battleConcluded)`** — **раунды**: после того как последний юнит в порядке хода (**`firstTeam`**, затем **`secondTeam`**) отыграл все фазы, цикл **начинается снова** с первого юнита и первой фазы, пока не объявлен исход боя (**Победа игрока** / **Победа компьютера** / **Ничья**) или пока раунд не окажется пустым (см. ниже).
4. **Исход боя** выставляется в анонс фазы (`currentPhaseName`) при изменении HP: живые юниты с `IsAI` — сторона компьютера, без `IsAI` — игрок. Если за раунд **ни один** юнит не прошёл фазы (у всех нет настроенных кубиков), цикл раундов **останавливается** с предупреждением в лог.

---

## 2. Начало «хода» юнита (перед фазами)

Для каждого корня юнита в порядке хода (**`GetUnitRootsInTurnOrder()`**: сначала **`firstTeam`**, затем **`secondTeam`**) **перед** входом в цикл по `battleActions`:

| Шаг | Что делается |
|-----|----------------|
| Проверка исхода | Если `battleConcluded` — выходим из циклов. |
| Пустой слот | `null` в списке — предупреждение, юнит пропускается. |
| Нет `Unit` | Предупреждение, пропуск. |
| Нет кубиков в модулях | `HasAtLeastOneDiceInModules() == false` — предупреждение, **весь ход юнита пропускается** (фазы для него не выполняются). |

**Отдельного «пре-фазового» скрипта нет** — это вся подготовка на стороне `BattleController` перед первой фазой.

---

## 3. Каждая фаза (общая оболочка `BattleController`)

Для каждого ненулевого `BattleActions` из списка:

1. **`currentPhaseName`** = `PhaseName` фазы (показывается в `OnGUI`: «Фаза: …»).
2. Выполняется **`yield return action.Action(battleController, unitIndex, unit)`** — вся логика фазы внутри.
3. Если за время фазы объявлен **`battleConcluded`** (например после урона по HP) — дальнейшие фазы этого юнита и следующие юниты в раунде не идут до следующего раунда не дойдут: внешний цикл завершится после выхода из вложенных циклов.
4. Если у фазы **`UsesManualAdvanceClick == true`** и юнит **не ИИ** — ожидание **ЛКМ** по окну игры (после фазы). У **EnergyRegen**, **ShieldUp** и **DamageDeal** это **`false`** — дополнительного клика от контроллера нет.
5. **`unit.ResetModuleDiceToLocalLayout()`** — у **всех** `Modules` под юнитом кубики из слотов возвращаются под локальные позиции модулей (сброс после фазы).

После **всех** фаз у этого юнита снова вызывается **`ResetModuleDiceToLocalLayout()`** (двойной сброс в конце хода — намеренно).

---

## 4. Правило раскладки кубиков на экране (общее)

Когда фаза вызывает **`LayoutDiceGroupCenteredOnScreen`** (или старый путь **`LayoutModuleDice`** по одному модулю):

- Точка привязки — **центр вьюпорта** камеры: `ViewportToWorldPoint(0.5, 0.5, diceViewDistanceFromCamera)`.
- Кубики выстраиваются в ряд вдоль **`camera.right`**.
- Расстояние между **центрами** соседних кубиков — **`diceHorizontalSpacing`**.
- Группа **центрируется** по горизонтали (как строка текста по центру).
- Родитель трансформа кубика **не меняется** — меняется только **мировая позиция** (`position`).

Это правило **не «проверяется» каждый кадр** — раскладка вызывается **явно** в начале показа группы и **после частичного расхода** кубиков там, где фаза перераскладывает оставшиеся (щит, турели).

---

## 5. Первая фаза в списке (типично **EnergyRegen — Восстановление энергии**)

Выполняется **только если** на юните есть хотя бы один **`MGenerator`** и после пополнения есть непустые кубики.

| Этап | Действие |
|------|----------|
| Пополнение слотов | На каждом `MGenerator`: **`ReplenishConsumedDice()`** — пустые слоты заполняются копиями по шаблонам. |
| Сбор кубиков | Все не-`null` `EnergyDice` (через `Dices`) попадают в общий список. |
| Раскладка | **`LayoutDiceGroupCenteredOnScreen(allDice)`** — правило из §4. |
| Бросок | Все кубики бросаются **параллельно** (`RollDice` в отдельных корутинах). |
| Результат | Для каждого кубика: если грань **не fail** и тип **`EnergyDice`** — начисление заряда на «свой» генератор по таблице грани. |
| Игрок | Если юнит не ИИ и кубики были — ожидание **ЛКМ** (внутри фазы), пока кубики ещё на экране. |
| Завершение | **`RemoveDice` + Destroy** корня слота для каждого выложенного кубика; затем **`ResetModuleDiceToLocalLayout()`** на юните (ещё раз и с контроллера после фазы). |

**Пропуска фазы нет** — либо выход сразу (нет генераторов / нет кубиков), либо полный сценарий выше.

---

## 6. Последующие фазы

### 6.1 **ShieldUp — Пополнение щита**

Условие входа: есть **`MShield`** с кубиками после **`ReplenishConsumedDice`** на всех щитах.

| Этап | Действие |
|------|----------|
| Пополнение | На каждом `MShield`: **`ReplenishConsumedDice()`**. |
| Раскладка | **`LayoutDiceGroupCenteredOnScreen`** всех кубиков щитов — §4. |
| Бросок | Параллельный бросок всех этих кубиков; во время броска учитывается **«Пропустить»**. |
| Игрок | Кнопка **`OnGUI` «Пропустить»**; ЛКМ по кубику щита (raycast), клик по кнопке пропуска не считается выбором кубика. |
| ИИ | По очереди берётся первый доступный кубик щита. |
| Выбор кубика | Стоимость/восстановление с **`ShieldDice`** по выпавшей грани; энергия списывается с **`MGenerator`** по тому же принципу, что и в коде щита; **`unit.AddShield`**. Кубик снимается и уничтожается. |
| После каждого выбора | Оставшиеся кубики снова **`LayoutDiceGroupCenteredOnScreen`** (§4). |
| Пропуск | В `finally`: все оставшиеся кубики щитов **Destroy** (слоты пустые). |

### 6.2 **DamageDeal — Нанесение урона**

Условие входа: есть **`MTurret`** с настроенными слотами кубиков (**`HasConfiguredDiceForBattle`**); экземпляры ещё не созданы.

| Этап | Действие |
|------|----------|
| Цель | Определяется атакующим: ИИ — автоматически; игрок при ≥2 противниках — клик по цели; при одном — без клика. |
| Пополнение | **После выбора цели** на каждом `MTurret`: **`ReplenishConsumedDice()`** (здесь появляются клоны кубиков). |
| Раскладка | **`LayoutDiceGroupCenteredOnScreen`** всех кубиков турелей — §4. |
| Бросок | Параллельный бросок; опционально **«Пропустить»** (как у щита). |
| Игрок / ИИ | Аналогично щиту: выбор кубика турели или пропуск; ИИ — по очереди. |
| Эффект выстрела | Если грань **не fail**: при достаточной энергии на генераторах списывается **`EnergyCostPerShot`** модуля, цели наносится **`BaseDamagePerShot`** (цель — следующий живой **противник** в порядке хода из двух команд). Урон по **`Unit.TakeDamage`**: **сначала щит, потом HP**. |
| Кубик | Всегда снимается и уничтожается после выбора. |
| После выбора | **`RelayoutRemainingTurretDice`** — снова §4. |
| Пропуск | Оставшиеся кубики турелей **Destroy**. |

### 6.3 Другие фазы в `battleActions`

Любой другой компонент **`BattleActions`** получает тот же контракт из §3. Если он сам вызывает **`LayoutModuleDice` / `LayoutDiceGroupCenteredOnScreen`**, к нему применяется §4. После фазы контроллер **всегда** делает **`ResetModuleDiceToLocalLayout`**.

---

## 7. Сводка: где что выравниваем

| Момент | Кто вызывает |
|--------|----------------|
| Показ группы кубиков в начале фазы | EnergyRegen / ShieldUp / DamageDeal → `LayoutDiceGroupCenteredOnScreen` |
| После частичного расхода кубиков на экране | ShieldUp, DamageDeal → повторная `LayoutDiceGroupCenteredOnScreen` для оставшихся |
| После **каждой** фазы (любой) | `BattleController` → `unit.ResetModuleDiceToLocalLayout()` |
| После **всех** фаз юнита | `BattleController` → ещё раз `ResetModuleDiceToLocalLayout()` |
